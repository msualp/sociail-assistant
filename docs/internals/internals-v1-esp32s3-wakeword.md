# Sociail Assistant (MagSafe) — Internals v1 Build Doc
**Goal:** On-gizmo wake word (device is always listening) → iPhone relays audio to Sociail backend for full speech recognition + intent.

**v1 demo reality:** use a **pre-trained WakeNet wake phrase** (recommended: **“Hi ESP”**) for the demo. The exact branded wake phrase **“Hey Sociail”** is a **v1.5** project (see the decision tree in Section 6).

**v1 definition:** The gizmo can wake the system without the phone running wake word detection.

> This doc is written for a first-time hardware builder: it’s verbose, step-by-step, and includes gotchas + success checks.

---

## 0) What “Done” Looks Like (Success Criteria)
You can attach the gizmo to the back of an iPhone (MagSafe), open the Sociail iPhone app, and reliably do:

- Say **"Hi ESP… status"** _(v1 demo phrase; branded "Hey Sociail" is v1.5)_
- Gizmo LED flips to **Listening**
- Phone captures and streams the utterance **until silence** to backend
- Backend replies with **workspace + room + humans/AIs + health + active task**
- Phone speaks the response (iPhone speaker)
- Gizmo confirms with LED pattern (workspace tint + health pulse)

Also:
- **Long-press button = STOP NOW** (immediate interrupt even if network is slow)
- Battery indicator shows **high/med/low** (demo-friendly, not perfect)

## 1) Architecture (Truth, Not Marketing)

### 1.1 Data flow (primary path: BLE → iPhone → backend)
1) **Gizmo** continuously processes mic audio locally for wake phrase (v1 demo: **"Hi ESP"**) using an embedded wake-word engine  
2) On detection: gizmo sends BLE event `WAKE_DETECTED` to iPhone  
3) iPhone begins (or continues) audio capture and streams audio to backend **until silence**  
4) Backend returns `{intent, workspace, room, presence, health, active_task, tts}`  
5) iPhone plays `tts` on iPhone speaker, and sends BLE `STATE/RESULT` to gizmo to drive LEDs

### 1.2 Secondary path (optional): Wi‑Fi direct
If the phone is absent, you can later add:
- gizmo connects to Wi‑Fi and hits the same backend endpoint directly (v1.5+)
- **do not build this in v1** unless you need it immediately

### 1.3 The two “wake word” strategies for v1
**Strategy A — ESP32‑S3 only (recommended first):** Espressif **WakeNet** in ESP‑SR  
- WakeNet supports up to 5 wake words per model and is designed for embedded always-on wake word detection.  
- Customization exists, but is a process.  
Refs: WakeNet overview and customization docs.

**Strategy B — Wake-word coprocessor (best low-power / clean product path):** Syntiant **NDP120** (e.g., Arduino Nicla Voice) triggers the ESP32‑S3  
- Extremely low power always-on keyword detection, ESP32‑S3 wakes only when triggered.  
Refs: Arduino Nicla Voice + NDP120 overview docs.

### 2.4 Power Budget

**Target battery life:** 12–16 hours active use (v1 demo goal)

| State | Est. Current | Duration | Notes |
|-------|--------------|----------|-------|
| **IDLE** (WakeNet running) | 35–50 mA | 90% of time | Mic + wake engine continuous |
| **LISTENING** (BLE active) | 80–100 mA | Bursts | Wake triggered; BLE connected; LEDs indicate listening. (Audio is captured/streamed by iPhone in v1.) |
| **THINKING** (LED chase) | 60–80 mA | Brief | Waiting for backend |
| **LEDs at full white** | 150–200 mA | Avoid | Cap brightness to 30% max |
| **Deep sleep** (future) | <1 mA | N/A | Requires NDP120 coprocessor |

**Battery sizing:**
- 1000 mAh @ 45 mA average = ~22 hours theoretical
- Real-world with bursts: ~14–18 hours
- **Recommendation:** 800–1000 mAh LiPo for v1 demo

**Reality check (measure this):** Seeed lists the XIAO ESP32S3 typical board consumption around **~22 mA on BAT** in a basic running state. Real v1 current depends heavily on LED brightness, wake engine configuration, and BLE connection interval. Plan to measure with a USB power meter and log current for each state.

**Measurement recipe (10 minutes):**
1) Set LEDs OFF, wake engine ON → record idle mA for 2 minutes.
2) Set `LISTENING` LED pattern at 30% brightness → record.
3) Set `THINKING` chase at 30% brightness → record.
4) Trigger 20 wake events in a row → confirm no brownouts.


**Power gotchas:**
- WS2812 LEDs at full brightness can pull 60 mA/LED × 8 = 480 mA (brownout risk)
- Always cap LED brightness in firmware: `brightness = min(brightness, 76)` (30%)
- ESP32-S3 Wi-Fi (if enabled) adds 100+ mA — keep Wi-Fi off in v1

### 2.5 Center Button Module (default 24mm, alternate 22mm)

**Goal:** a single, large, “premium-feel” center button that is thumb-sized **and** does **not** get accidentally pressed when the iPhone (with gizmo attached) is placed back-down on a table.

#### Compact rationale
- **24mm (default):** easier thumb hit rate, stronger demo feel, better with gloves.
- **22mm (alternate):** subtler look and slightly less accidental-contact risk.

#### Strategy (fast + controllable)
We separate **feel** (mechanics) from **electrical reliability** (switch):
- **Switch (electrical):** Adafruit **Soft Silicone Top 6mm** tact switch (soft/grippy top, consistent electrical behavior).
- **Plunger (feel):** 3D printed **TPU 95A** concave plunger that enlarges the tact switch into a single “thumb button”.
- **Feel tuning:** add **one** of:
  - a small **~1mm foam disk** between plunger stem and switch (softer + quieter), or
  - a tiny **6mm silicone bumper dot** (elastomer nub; snappy + damped)

#### Preventing accidental presses (non-negotiable)
Concavity helps, but the real protection is geometry:
- Add a **guard bezel** around the button that is **0.8–1.2mm taller** than the highest point of the plunger rim.
- Seat the plunger so its top surface sits **0.3–0.7mm below** the bezel height.
- Firmware gating: require **press-and-hold 150–250ms** to register a press.
- **Bezel material:** the guard bezel should be **rigid** (PLA/PETG) for true anti-press behavior; TPU bezels can compress and still trigger the switch.

#### What we will build
Two interchangeable test articles:
- **Prototype A:** 22mm concave TPU plunger + bezel guard
- **Prototype B:** 24mm concave TPU plunger + bezel guard

#### Mechanical targets (starting values)
- Plunger OD: **22.0mm** (A) / **24.0mm** (B)
- Cutout/hole in top shell: **OD + 0.6mm** (clearance)
- Bezel OD: **plunger OD + 4mm**
- Bezel height above shell: **~1.0mm**
- Plunger top sits: **~0.5mm below bezel top**
- Button travel: **~0.8–1.2mm** total

#### Print guidance
- **TPU plungers:** TPU 95A, 0.16–0.20mm layers, 3–4 walls, 15–25% gyroid, slow TPU profile
- **Bezel guards:** PLA/PETG (or TPU), 0.16–0.20mm layers, 3 walls, 10–15% infill

#### Prototype assets
Canonical module assets:
- `prototype/modules/button-module-22-24-v1/`
- `prototype/modules/button-module-22-24-v1.zip`

Contains: plungers, rigid bezel guards, and negative cutouts for Bambu Studio

Assembly summary:
1) Use the `neg_button_cutout_XXmm.stl` as a **Negative Part** in Bambu Studio to cut the top-shell opening.
2) Print the TPU plunger.
3) Print (or temporarily glue) the bezel guard.
4) Mount the tact switch under the opening so the plunger stem contacts it.
5) Add foam disk or silicone nub to tune feel.

| Microphone | ✓ Built-in PDM mic | ✓ |

| RAM for WakeNet | ✓ 8MB PSRAM | ✓ |
| Flash | ✓ 8MB | ✓ |
| Battery charging | ✓ Built-in | ✓ |
| Form factor | ✓ 21×17.5mm | ✓ |

**Bonus:** OV2640 camera included — enables future "Hey Sociail, what's this?" visual queries.

**Important Sense gotchas (save yourself hours):**
- The **mic/camera/microSD** live on the **Sense expansion board**. Make sure it’s fully seated (press until you hear/feel a *click*).
- The onboard mic uses **GPIO42 (PDM CLK)** and **GPIO41 (PDM DATA)**.
- If you use the microSD feature, it consumes pins used by the expansion board (notably **GPIO3/7/8/10**). Avoid reusing those for your own wiring.
- On ESP32‑S3, the pins labeled A11/A12 (GPIO41/GPIO42) should not be treated as ADC inputs.

**Core components:**
- XIAO ESP32S3 Sense (with camera + mic)
- WS2812/NeoPixel LED strip (8 LEDs)
- Tactile button (6×6mm momentary)
- LiPo battery (3.7V, 800–1000 mAh)
- Battery connection: **solder battery leads** (or a 1.25mm pigtail) to the XIAO **BAT+/BAT- pads** (Seeed docs show battery solder pads; no dedicated battery ADC pin)
- MagSafe magnet ring (adhesive-backed)

**Nice-to-have:**
- MAX17048 fuel gauge (I2C) for accurate battery %
- Slide switch for true power off

### Variant 2 (Product-grade always-on): ESP32‑S3 + NDP120 coprocessor
Use if you want always-on wake word without draining battery.

**Adds:**
- NDP120 module / board (e.g., Arduino Nicla Voice) for wake word / keyword spotting
- I2C/UART trigger line into ESP32‑S3 (simple interrupt-style wake)

**Why it's good:**
- ESP32 sleeps most of the time
- Wake word runs in ultra-low power silicon (<1 mA)

### 4.0 XIAO ESP32S3 / Sense Pin Mapping (use this, not guesswork)

**Physical pin label → ESP32-S3 GPIO**

| XIAO Pin | GPIO | Notes |
|---|---:|---|
| D0 | 1 | Safe general IO / ADC |
| D1 | 2 | Safe general IO / ADC |
| D2 | 3 | ADC; also used by Sense expansion for microSD CS in some configs |
| D3 | 4 | Safe general IO / ADC (good candidate for battery divider input) |
| D4 (SDA) | 5 | I2C SDA / ADC |
| D5 (SCL) | 6 | I2C SCL / ADC |
| D6 (TX) | 43 | UART TX; **strapping-related** — avoid if you don’t need it |
| D7 (RX) | 44 | UART RX; **strapping-related** — avoid if you don’t need it |
| D8 (SCK) | 7 | SPI SCK; often used by Sense expansion microSD |
| D9 (MISO) | 8 | SPI MISO; often used by Sense expansion microSD |
| D10 (MOSI) | 9 | SPI MOSI |
| D11 | 42 | **PDM MIC CLK** when mic enabled; do not rely on ADC here |
| D12 | 41 | **PDM MIC DATA** when mic enabled; do not rely on ADC here |

**Warnings that prevent painful bugs:**
- If the Sense mic is enabled, **GPIO41/42 are consumed by the mic**.
- Some pins are used by the Sense expansion (camera/microSD). If you enable microSD, expect **GPIO3/7/8/10** to be busy.
- Avoid using BOOT (GPIO0) for anything in v1.

### 4.4 Battery Measurement (you asked for battery-led)

**Important:** Seeed states the **XIAO ESP32S3 series (including Sense)** has **no GPIO configured for battery voltage measurement** out of the box. If you want battery level, you must add hardware (MAX17048) or a divider. (Note: a different product, **XIAO ESP32S3 Plus**, may expose an `ADC_BAT` pin — do not confuse the two.)

You have two practical options:

**Option A (recommended): MAX17048 fuel gauge (accurate + easy)**
- Wire MAX17048 to I2C (D4/D5) and read percent + voltage.

| Fuel Gauge | XIAO Pin | GPIO | Notes |
|---|---|---:|---|
| SDA | D4 | 5 | I2C data |
| SCL | D5 | 6 | I2C clock |
| VCC | 3V3 | — | 3.3V |
| GND | GND | — | |

**Option B (fallback): resistor divider to an ADC pin (good enough for demo)**
- Parts: **2×220k** resistors (VBAT→ADC and ADC→GND) + optional **0.1µF** cap from ADC→GND.
- Use an ADC-capable pin that isn’t consumed by the Sense expansion. A good default is **D3 (GPIO4)** *only if you are not using microSD/camera features that claim GPIO4 in your firmware config*. If in doubt, use MAX17048.

**Wiring:**
- VBAT+ → 220k → ADC pin (D3/GPIO4)
- ADC pin → 220k → GND
- (Optional) 0.1µF from ADC pin → GND to smooth noise

**Voltage math:** divider halves the voltage, so:
- `VBAT ≈ 2 × Vadc`

**Simple battery buckets (demo-friendly):**
- >= 4.0V: HIGH
- 3.7–4.0V: MED
- <= 3.7V: LOW

**Gotcha:** don’t promise true % with a divider; show HIGH/MED/LOW and you’re fine.

## 5) Firmware (ESP32‑S3) — State Machine + Tasks

### 5.1 States
- `IDLE` (breathe + workspace tint + battery pulse every 10s)
- `LISTENING` (solid white)
- `THINKING` (chase)
- `OK` (workspace-tinted blink)
- `ERR` (red blink)

### 5.2 Tasks (FreeRTOS)
- `audio_task`: reads I2S frames into a ring buffer
- `wake_task`: runs wake word detection on frames; on detection triggers event
- `ble_task`: BLE GATT service notify/write handling
- `led_task`: LED animations (non-blocking)
- `battery_task`: reads ADC or fuel gauge; publishes battery updates

### 5.3 Device → Phone BLE messages (control channel)
Keep messages short and consistent:
- `WAKE_DETECTED`
- `BTN_SHORT`
- `BTN_LONG` (hard stop)
- `BATT_MV: ####` (or percent)


### 5.4 Phone → Device BLE messages
- `STATE: idle|listening|thinking|ok|err`
- `WORKSPACE: work|personal|family`
- Optional: `HEALTH: green|yellow|red`

### 5.4.1 BLE UUIDs + GATT Schema (exact)

**Goal:** keep BLE *boringly reliable* — short messages, low rate, clear ownership.

#### Service + characteristics
Use **one custom service** plus **two characteristics**.

- **Service UUID (128-bit):** `8A9B0001-2C4D-4E5F-9A0B-1C2D3E4F5001`

**Characteristic A — CTRL (events + commands)**
- **UUID:** `8A9B0002-2C4D-4E5F-9A0B-1C2D3E4F5001`
- **Properties:** `Notify` (device→phone), `WriteWithoutResponse` (phone→device)
- **Use for:** wake/button events, state changes, workspace tint, stop/cancel

**Characteristic B — TELEMETRY (battery + diagnostics)**
- **UUID:** `8A9B0003-2C4D-4E5F-9A0B-1C2D3E4F5001`
- **Properties:** `Notify` (device→phone)
- **Use for:** battery mV/% (MAX17048), RSSI, last wake confidence, error codes

> Keep it to 2 characteristics in v1. More is how BLE projects get flaky.

#### Payload format (v1)
Use **compact JSON UTF-8** for speed of iteration.

Rules:
- Keep each message **< 180 bytes** (safe even when MTU is small)
- Rate limit notifications:
  - CTRL: **<= 10/sec** burst
  - TELEMETRY: **1 every 10s** (battery), **1 on error**, **1 on connect**
- Always include:
  - `t` (type)
  - `seq` (monotonic int)
  - `ts` (ms since boot)

##### Device → Phone (Notify on CTRL)
```json
{"t":"wake","seq":101,"ts":123456,"conf":0.84}
```
```json
{"t":"btn","seq":102,"ts":123789,"v":"short"}
```
```json
{"t":"btn","seq":103,"ts":124200,"v":"long"}
```

##### Phone → Device (WriteWithoutResponse on CTRL)
```json
{"t":"state","seq":2201,"ts":55321,"v":"listening"}
```
```json
{"t":"workspace","seq":2202,"ts":55340,"v":"personal"}
```
```json
{"t":"health","seq":2203,"ts":55360,"v":"green"}
```
```json
{"t":"stop","seq":2204,"ts":55380,"reason":"user"}
```

##### Device → Phone (Notify on TELEMETRY)
```json
{"t":"batt","seq":301,"ts":130000,"mv":3870,"pct":62}
```
```json
{"t":"diag","seq":302,"ts":131000,"wake_fp":0,"wake_fn":1,"ble_rssi":-62}
```

#### Connection behavior (non-negotiable)
- If the phone disconnects mid LISTENING/THINKING: gizmo **returns to IDLE** within 1s.
- The phone is the UI/stream controller. The gizmo never “keeps listening forever” waiting for the phone.
- Button long-press is a hard interrupt: phone should stop recording + send backend cancel.

#### Optional: binary TLV later (v2)
If JSON ever becomes a problem, migrate CTRL payloads to a tiny TLV:
- Byte0 = type
- Byte1 = flags
- Byte2-3 = seq
- Remaining = payload

But do **not** do this in v1.

### 5.5 Timeout & Error Recovery

The gizmo must handle failure gracefully without user intervention.

**State timeout rules:**

| State | Timeout | Action on Timeout |
|-------|---------|-------------------|
| `LISTENING` | 15s | → `IDLE` + single red blink |
| `THINKING` | 20s | → `ERR` for 2s → `IDLE` |
| `OK` | 3s | → `IDLE` (normal completion) |
| `ERR` | 5s | → `IDLE` |

**BLE disconnect handling:**

```
On BLE disconnect:
  1. If in LISTENING or THINKING:
     - Immediately → IDLE
     - Triple red blink (indicates phone lost)
  2. If in IDLE:
     - Continue normal idle breathing
     - Attempt reconnect passively (BLE advertising)
  3. Wake word still active regardless of BLE state
```

**Reconnection behavior:**
- Gizmo always advertises when not connected
- On reconnect: send `BATT_MV` and current state
- No automatic retry spam — let phone initiate

**Hard reset:**
- Long-press button (3s) while in any error state → force `IDLE`
- Long-press button (5s) → restart device (last resort)

**Implementation skeleton:**
```c
// timeout_task (pseudo)
void timeout_check() {
  uint32_t now = millis();
  switch (current_state) {
    case LISTENING:
      if (now - state_entered_at > 15000) {
        led_blink_red(1);
        set_state(IDLE);
      }
      break;
    case THINKING:
      if (now - state_entered_at > 20000) {
        set_state(ERR);
      }
      break;
    // ...
  }
}
```

## 6) Wake Word Implementation (v1)

### 6.1 WakeNet (ESP‑SR) path (recommended to start)
WakeNet is an embedded wake word engine intended for always-on use. It supports up to 5 wake words per model and has a documented customization process.
- WakeNet model doc: https://docs.espressif.com/projects/esp-sr/en/latest/esp32s3/wake_word_engine/README.html
- Customization process: https://docs.espressif.com/projects/esp-sr/en/latest/esp32s3/wake_word_engine/ESP_Wake_Words_Customization.html
- ESP‑SR component registry: https://components.espressif.com/components/espressif/esp-sr

### 6.3 Wake Word Tuning Protocol

Wake word detection is a tradeoff between **false positives** (triggers when it shouldn't) and **false negatives** (misses when it should trigger).

**Target metrics for v1:**

| Metric | Target | Acceptable | Notes |
|--------|--------|------------|-------|
| False positive rate | <1/hour | <3/hour | In typical home/office ambient |
| False negative rate | <5% | <10% | When spoken clearly at 1m |
| Trigger latency | <300ms | <500ms | From end of phrase to LED change |

**Tuning procedure:**

**Step 1: Baseline false positive test**
1. Place device in typical environment (desk, living room)
2. Run for 2 hours with TV/music/conversation in background
3. Log all triggers with timestamp
4. Target: <2 false triggers in 2 hours

**Step 2: True positive test**
1. Say wake phrase 20 times at 1 meter distance
2. Vary: standing, sitting, different speakers if possible
3. Log hits and misses
4. Target: 18+ / 20 triggers (90%+)

**Step 3: Adjust threshold**
- WakeNet has a `detection_threshold` parameter (typically 0.5–0.95)
- Higher threshold = fewer false positives, more false negatives
- Start at 0.8, adjust based on test results

**Step 4: Add cooldown**
```c
#define WAKE_COOLDOWN_MS 2000  // Ignore triggers for 2s after last trigger

if (wake_detected && (now - last_wake_time > WAKE_COOLDOWN_MS)) {
  last_wake_time = now;
  ble_notify("WAKE_DETECTED");
}
```

**Step 5: Document results**
Keep a tuning log:
```
Date: 2025-03-04
Threshold: 0.82
FP test: 1 trigger in 2h (pass)
TP test: 19/20 (95%, pass)
Notes: Missed one quiet utterance from 2m away
```

**Goal:** move from a demo wake phrase (“Hi ESP”) to a branded wake phrase (“Hey Sociail”) without derailing shipping.

#### Path A — Pay Espressif to train WakeNet (highest confidence, slowest)

- Cons: not truly “device wakes phone” unless iOS stays foreground/background permissions allow.

#### Path D — Wake coprocessor (product-grade always-on)
- Use NDP120 (or similar) for always-on wake word.
- Pros: power efficient + “real product” behavior.
- Cons: additional hardware + integration time.

**Recommendation:** start with Path B exploration while collecting data for Path A. If battery life matters, plan Path D.

### 7.1 PDM Mic Capture (XIAO ESP32S3 Sense)

The XIAO ESP32S3 Sense has a **built-in PDM microphone**. Use PDM mode, not standard I2S.

Espressif I2S/PDM docs: https://docs.espressif.com/projects/esp-idf/en/v5.0/esp32s3/api-reference/peripherals/i2s.html

**WakeNet AFE gotcha:** WakeNet runs on top of Espressif’s AFE pipeline, which expects a specific audio format (typically 16 kHz, mono) and a correct front-end configuration. Even if you can record audio successfully, wake detection can still fail if AFE config, channel mode, or scaling is off. If you hit this, don’t “keep guessing” — follow the De-risk plan in 6.1.

**Fast sanity test (Arduino-style):**
If you just want to confirm the mic works before going deep on ESP-IDF, many examples use:

```cpp
// PDM pins for XIAO ESP32S3 Sense mic
i2s.setPinsPdmRx(42, 41);
```

If you can see the mic level move in Serial Plotter, your hardware is good.

```c
// pdm_audio.c (skeleton for XIAO ESP32S3 Sense)
#include "driver/i2s.h"

#define I2S_PORT I2S_NUM_0
#define SAMPLE_RATE 16000
#define BUF_SAMPLES 512

// XIAO ESP32S3 Sense internal PDM mic pins
#define PDM_CLK_PIN  42
#define PDM_DATA_PIN 41

void pdm_init_mic() {
  i2s_config_t cfg = {
    .mode = I2S_MODE_MASTER | I2S_MODE_RX | I2S_MODE_PDM,  // PDM mode!
    .sample_rate = SAMPLE_RATE,
    .bits_per_sample = I2S_BITS_PER_SAMPLE_16BIT,
    .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,
    .communication_format = I2S_COMM_FORMAT_STAND_I2S,
    .intr_alloc_flags = ESP_INTR_FLAG_LEVEL1,
    .dma_buf_count = 4,
    .dma_buf_len = BUF_SAMPLES,
    .use_apll = false
  };

  i2s_pin_config_t pins = {
    .bck_io_num = I2S_PIN_NO_CHANGE,   // Not used in PDM
    .ws_io_num = PDM_CLK_PIN,          // PDM clock
    .data_out_num = I2S_PIN_NO_CHANGE,
    .data_in_num = PDM_DATA_PIN        // PDM data
  };

  ESP_ERROR_CHECK(i2s_driver_install(I2S_PORT, &cfg, 0, NULL));
  ESP_ERROR_CHECK(i2s_set_pin(I2S_PORT, &pins));
  i2s_zero_dma_buffer(I2S_PORT);
}

size_t pdm_read_frame(int16_t *dst, size_t samples) {
  size_t bytes_read = 0;
  i2s_read(I2S_PORT, dst, samples * sizeof(int16_t), &bytes_read, portMAX_DELAY);
  return bytes_read / sizeof(int16_t);
}
```

**Alternative: External I2S Mic (INMP441)**

If the built-in PDM mic quality is insufficient, use an external I2S mic:

```c
// i2s_audio.c (for external INMP441 on D8/D9/D10 — avoids UART/strapping pins)
void i2s_init_external_mic() {
  i2s_config_t cfg = {
    .mode = I2S_MODE_MASTER | I2S_MODE_RX,  // Standard I2S
    .sample_rate = 16000,
    .bits_per_sample = I2S_BITS_PER_SAMPLE_32BIT,  // INMP441 outputs 24-bit in 32-bit frame
    .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,
    .communication_format = I2S_COMM_FORMAT_I2S,
    .dma_buf_count = 4,
    .dma_buf_len = 512,
  };

  i2s_pin_config_t pins = {
    .bck_io_num = GPIO_NUM_7,    // D8 (SCK)  → BCLK
    .ws_io_num  = GPIO_NUM_8,    // D9 (MISO) → LRCLK
    .data_in_num = GPIO_NUM_9,   // D10 (MOSI) → DOUT
  };
  // ... install and set pins
}
```

> **Warning:** D8/D9/D10 are also used by the onboard microSD slot. If you need microSD, reassign INMP441 to other free GPIOs and update this map.
```

**Gotchas**
- PDM vs I2S: the built-in mic is PDM — use `I2S_MODE_PDM` flag
- If you "hear nothing," check that PDM pins match (GPIO41/42)
- INMP441 outputs 24-bit data in 32-bit frames — shift right by 8 bits

**Success check**
- Dump RMS level over UART and see it change when you speak
- Target: ~500–2000 RMS for speech at 0.5m, <200 for silence

## 8) iPhone App (Relay) — v1 Changes

In v0 you may run wake word on iPhone.
In v1, gizmo wakes the phone app via BLE event.

### 8.1 Avoid missing the first words after wake
If you start recording only after WAKE_DETECTED, you can miss the first syllable.

**Best v1 approach (foreground app):**
- iPhone keeps a short ring buffer of audio (e.g., last 0.8s) while app is open
- When WAKE_DETECTED arrives, phone starts streaming immediately and includes the pre-roll buffer

### 8.2 "Until silence" VAD
- Use energy/RMS threshold
- Stop after ~1000ms below threshold
- Always stop immediately on BTN_LONG

### 8.3 iOS Background Limitations (Important)

**The ring buffer approach (Section 8.1) only works when the app is in foreground.**

**Why:**
- iOS aggressively suspends backgrounded apps
- Backgrounded apps cannot maintain continuous mic access
- BLE notifications may be delayed 1–10 seconds when app is suspended

**What this means for v1:**
- User must have Sociail app **open and visible** for best experience
- If app is backgrounded: wake word still triggers, but first ~0.5s of speech may be lost
- No pre-roll buffer when backgrounded

**Demo script requirement (v1):** before you demo voice, **open Sociail and keep it foregrounded**, then attach the MagSafe gizmo. Your demo flow should explicitly include: “Open Sociail → attach gizmo → say wake phrase → speak command.” This avoids awkward failures.

**Mitigation options (choose one for v1):**

| Option | Tradeoff | Recommendation |
|--------|----------|----------------|
| **A. Accept limitation** | Document that foreground is required | ✓ Use for v1 |
| B. Background audio mode | Battery drain, App Store scrutiny | Defer to v1.5 |
| C. Stream audio from gizmo | BLE bandwidth limit (~20 kbps), latency | Not viable |

**User guidance (add to app):**
- Show reminder: "Keep Sociail open for best voice response"
- Consider iOS widget for quick return to app

**v1.5 path:**
- Investigate `BGTaskScheduler` for periodic wake
- Consider Audio Session background mode (requires justification to Apple)
- NDP120 coprocessor could reduce reliance on phone pre-roll

### 8.4 iOS Relay Checklist (CoreBluetooth + ring buffer + VAD + WebSocket)

This is the “phone side” bring-up checklist. Treat it like a test plan.

#### A) CoreBluetooth (device connection)
- [ ] Scan shows the gizmo within 10 seconds.
- [ ] Connect succeeds and stays stable for 2 minutes.
- [ ] Discover service `8A9B0001-2C4D-4E5F-9A0B-1C2D3E4F5001`.
- [ ] Subscribe to CTRL notifications (Characteristic A).
- [ ] Subscribe to TELEMETRY notifications (Characteristic B).
- [ ] Send a `STATE=idle` writeWithoutResponse on connect.

**Success:** You can tap the gizmo button and see `BTN_SHORT/BTN_LONG` events in the iOS log.

#### B) Audio ring buffer (pre-roll)
- [ ] Start mic capture while the app is foreground.
- [ ] Maintain a ring buffer of **0.8 seconds** of audio at 16 kHz PCM.
- [ ] On `WAKE_DETECTED`, include the pre-roll audio in the stream.

**Success:** "Hi ESP… status" does not clip the first word.

#### C) VAD (until silence)
- [ ] Compute RMS/energy per 20ms frame.
- [ ] Define `silence_threshold` after observing real noise floor.
- [ ] Stop after **~1000ms** below threshold.
- [ ] Hard-stop immediately on gizmo `BTN_LONG`.

**Success:** natural speech stops within ~1s after you finish speaking.

#### D) WebSocket streaming
- [ ] Connect to backend `wss://.../voice`.
- [ ] Send `start` JSON.
- [ ] Stream binary PCM frames (20ms) steadily.
- [ ] Send `stop` or `cancel`.
- [ ] Handle `final` or `error`.

**Success:** transcript+intent returns within ~1–2s after end of utterance.

#### E) UX wiring (state → LEDs)
- [ ] On WAKE: set gizmo `STATE=listening`.
- [ ] While streaming: keep `STATE=listening`.
- [ ] After stop: set `STATE=thinking`.
- [ ] On final: set `WORKSPACE=...`, `HEALTH=...`, then `STATE=ok`.
- [ ] On error: set `STATE=err` then back to `idle`.

**Success:** Gizmo LEDs always match what the user perceives.

#### F) Three acceptance tests (run 5× each)
- [ ] "Hi ESP… Status"
- [ ] "Hi ESP… Switch to Personal Workspace"
- [ ] "Hi ESP… Stop" (plus long-press STOP)

**Success:** 15/15 clean runs.

### 9.4 Backend WebSocket Contract (start/audio/stop/cancel)

**Goal:** one simple streaming contract that you can keep forever.

#### Transport
- **WebSocket** (wss) from iPhone → Sociail backend
- Audio is sent as **binary frames**; control messages are **JSON text frames**.

#### Audio format (v1)
- **PCM S16LE**, mono, **16 kHz**
- Chunk size: **20ms** (recommended) or **40ms**
  - 20ms @ 16kHz = 320 samples = 640 bytes
  - 40ms @ 16kHz = 640 samples = 1280 bytes

> Keep chunks small and regular. Bursty uploads cause lag and worse ASR.

#### Message types

##### 1) `start` (JSON text frame)
Send immediately when the phone enters LISTENING.

```json
{
  "type": "start",
  "session_id": "<uuid>",
  "user_id": "<user>",
  "device_id": "gizmo-001",
  "source": "iphone",
  "audio": {"sr": 16000, "fmt": "pcm_s16le", "ch": 1, "frame_ms": 20},
  "context": {"workspace": "server_authoritative"},
  "client": {"app": "sociail-ios", "ver": "0.1"}
}
```

##### 2) `ack` (JSON text frame)
Backend replies quickly to confirm session creation and return the authoritative workspace/room.

```json
{
  "type": "ack",
  "session_id": "<uuid>",
  "workspace": {"id": "work", "name": "Work"},
  "room": {"id": "room_123", "name": "Demo Room"}
}
```

##### 3) `audio` (binary frames)
Immediately after `start`, stream raw PCM frames as **binary** WebSocket messages.

Rules:
- No JSON wrapper on audio frames.
- Keep frames aligned to your chosen `frame_ms`.

##### 4) `stop` (JSON text frame)
Send when iPhone VAD detects silence (~1s below threshold) **or** user presses STOP.

```json
{"type":"stop","session_id":"<uuid>"}
```

##### 5) `cancel` (JSON text frame)
Send if the user hard-stops (gizmo long-press) or the app is interrupted.

```json
{"type":"cancel","session_id":"<uuid>","reason":"device_stop"}
```

##### 6) `final` (JSON text frame)
Backend returns final transcript + intent + authoritative state + optional TTS.

```json
{
  "type": "final",
  "session_id": "<uuid>",
  "transcript": "switch to personal workspace",
  "intent": {"name": "switch_workspace", "arg": "personal"},
  "workspace": {"id": "personal", "name": "Personal"},
  "room": {"id": "room_123", "name": "Home Office"},
  "presence": {
    "humans": [{"id":"u_mustafa","display":"Mustafa","state":"active"}],
    "ais": [{"id":"ai_sociail","display":"Sociail","state":"active"}]
  },
  "health": {"state":"green"},
  "active_task": {"state":"none", "label": null},
  "tts": "Switched to Personal workspace."
}
```

#### Status intent schema (locked)
When intent is `status`, backend **must** return these fields (even if empty):
- `workspace` + `room`
- `presence.humans[]` + `presence.ais[]`
- `health.state` in {`green`,`yellow`,`red`}
- `active_task.state` in {`none`,`running`} and `label`

#### Error handling
If backend fails, return:

```json
{"type":"error","session_id":"<uuid>","code":"ASR_DOWN","msg":"ASR unavailable"}
```

Phone should:
- speak a short error (“Backend unavailable”) and
- set gizmo LED to `ERR` for 2s → `IDLE`.
- Idle tint = workspace
- Status response: two-beat confirm:
  1) health color pulse
  2) task pulse (none=single blink, running=double pulse)

### 9.5 Backend Implementation Skeleton (so nobody misinterprets the contract)

This is intentionally “boring server code.” The objective is clarity.

#### Minimal responsibilities
- Maintain a **session state** keyed by `session_id`:
  - `user_id`, `device_id`, `workspace_id`, `room_id`, `started_at`
  - optional: `presence`, `health`, `active_task`
- Accept a WebSocket connection per session.
- Collect/stream audio frames until `stop` or `cancel`.
- On `stop`: run ASR → intent → update workspace/room state → return `final`.
- On `cancel`: stop ASR and release resources immediately.

#### Suggested state store interface
Use in-memory for v1, Redis later.

```python
class SessionStore:
    def create(self, session_id: str, state: dict) -> None: ...
    def get(self, session_id: str) -> dict | None: ...
    def update(self, session_id: str, patch: dict) -> None: ...
    def delete(self, session_id: str) -> None: ...
```

#### WebSocket handler pseudo-code

```python
async def ws_voice_handler(ws):
    session_id = None
    audio_frames = []  # v1: accumulate; v1.5: stream to ASR incrementally
    max_audio_bytes = 15 * 16000 * 2  # 15s cap @ 16kHz mono s16le
    audio_bytes = 0

    async for msg in ws:
        if msg.is_text():
            data = json.loads(msg.text())
            t = data.get("type")

            if t == "start":
                session_id = data["session_id"]
                # 1) load authoritative workspace/room for this user
                ws_state = await load_workspace_state(user_id=data["user_id"], device_id=data.get("device_id"))
                store.create(session_id, {**data, **ws_state, "started_at": now_ms()})
                await ws.send_json({
                    "type": "ack",
                    "session_id": session_id,
                    "workspace": ws_state["workspace"],
                    "room": ws_state["room"],
                })

            elif t == "stop":
                st = store.get(data["session_id"]) or {}
                pcm = b"".join(audio_frames)

                transcript = await asr_transcribe(pcm, sr=16000)
                intent = await intent_parse(transcript, st)

                # update authoritative session/workspace as needed
                st2 = await apply_intent_and_update_state(st, intent)

                # build final payload (must include locked status fields)
                final = build_final_payload(session_id=data["session_id"], transcript=transcript, intent=intent, state=st2)
                await ws.send_json(final)

                store.delete(data["session_id"])  # v1: one-shot sessions
                break

            elif t == "cancel":
                # stop everything immediately
                sid = data.get("session_id")
                store.delete(sid)
                break

            else:
                await ws.send_json({"type":"error","session_id":data.get("session_id"),"code":"BAD_TYPE","msg":"Unknown message type"})

        else:
            # binary frame: raw PCM chunk
            if session_id is None:
                continue  # ignore until start
            chunk = msg.bytes()
            audio_bytes += len(chunk)
            if audio_bytes > max_audio_bytes:
                await ws.send_json({"type":"error","session_id":session_id,"code":"AUDIO_TOO_LONG","msg":"Audio exceeded 15s cap"})
                store.delete(session_id)
                break
            audio_frames.append(chunk)

```

#### Notes (v1 vs v1.5)
- **v1**: it’s okay to accumulate frames until `stop`.
- **v1.5**: stream frames into the ASR engine as they arrive (lower latency, lower RAM).
- Put a hard cap on buffered audio in v1 (e.g., 15s) to prevent runaway memory usage.

## 10) Bring-up Checklist (Do this, check that)

This is the fastest path to first success. **Do not proceed** to the next checklist until the current one passes **10/10**.

### 10.1 Power + Boot
- [ ] Plug in USB-C. Board powers on reliably.
- [ ] No random resets for 2 minutes.
- [ ] If using LiPo: device boots on battery and on USB.

**If it fails:** remove LEDs, run USB-only, and confirm stable power first.

**Success:** stable boot + stable idle.

### 10.2 Flash + Serial Logging
- [ ] You can flash firmware repeatedly without drama.
- [ ] Serial console shows a boot banner and your firmware version.
- [ ] You print `state=IDLE` once per second (or similar heartbeat).

**Success:** reliable flashing + readable logs.

### 10.3 LED Bring-up
- [ ] Solid color works (white, red, green).
- [ ] Brightness cap works (<=30%).
- [ ] Idle breathe pattern runs for 2 minutes without resets.

**If it fails:** add 330Ω on DIN + 470–1000µF cap across LED power, shorten wires.

**Success:** stable LEDs, no flicker, no brownouts.

### 10.4 Button Bring-up
- [ ] Short press emits `BTN_SHORT` 10/10.
- [ ] Long press (>=0.8s) emits `BTN_LONG` 10/10.
- [ ] Debounce prevents double-trigger.

**Success:** button is boring and reliable.

### 10.5 Battery Telemetry Bring-up
Pick one:

**A) MAX17048 (recommended)**
- [ ] I2C scan finds the device.
- [ ] You can read voltage + percent.
- [ ] Battery notification is sent every 10s.

**B) Divider (fallback)**
- [ ] ADC reads a stable voltage.
- [ ] VBAT math is correct (`VBAT ≈ 2×Vadc`).
- [ ] You show HIGH/MED/LOW, not fake percent.

**Success:** battery display is stable and demo-safe.

### 10.6 BLE Bring-up (Device ↔ iPhone)
- [ ] iPhone sees the gizmo within 10 seconds.
- [ ] Connect stays stable for 2 minutes.
- [ ] Phone can set LED state instantly (`STATE=listening`).
- [ ] Device can notify `BTN_SHORT` and phone receives it.

**If it fails:** rate-limit notifications; lower LED brightness.

**Success:** stable BLE + bidirectional messages.

### 10.7 Mic Bring-up (PDM)
- [ ] Run Arduino mic sanity test OR print RMS level via ESP-IDF.
- [ ] RMS changes clearly when you speak.
- [ ] Noise floor is low enough that RMS is stable in silence.

**Success:** mic is real and usable.

### 10.8 Wake Word Bring-up
- [ ] Wake phrase triggers 18+/20 in a quiet room.
- [ ] False triggers are <1/hour.
- [ ] After wake: device notifies `WAKE_DETECTED` to phone.
- [ ] Cooldown prevents back-to-back accidental triggers.

**Success:** wake word is reliable enough to demo.

### 10.9 iPhone Relay Bring-up (Foreground)
- [ ] On `WAKE_DETECTED`, phone sets `STATE=listening`.
- [ ] Phone captures audio and streams to backend.
- [ ] Phone stops on silence (~1s below threshold).
- [ ] Phone plays TTS on iPhone speaker.

**Success:** a full wake→stream→reply loop works.

### 10.10 End-to-End Acceptance (3 core commands)
Run these three flows 5 times each:
- [ ] "Hi ESP… **Status**"
- [ ] "Hi ESP… **Switch to Personal Workspace**"
- [ ] "Hi ESP… **Stop**" (also test long-press STOP)

**Success:** 15/15 clean runs.

## 11) Gotchas (the stuff that burns time)
- **I2S mic**: pin mapping/bit depth mismatch is the #1 failure
- **Wake**: tune sensitivity; add a short cooldown after triggers
- **LED power**: cap brightness (WS2812 can brown out small boards)
- **MagSafe**: camera bump interference may require slight offset in v1.1

### 11.1 Troubleshooting Matrix (symptom → likely cause → fix)

| Symptom | Likely cause | Fix (do this, in order) | Success check |
|---|---|---|---|
| iPhone can’t find the gizmo in BLE scan | Not advertising, crashed, wrong service UUID | 1) Reboot gizmo 2) Confirm advertising is on 3) Verify Service UUID matches 4) Turn Bluetooth off/on on iPhone | Gizmo appears within 5–10s |
| iPhone connects then disconnects repeatedly | Too many notifications, invalid GATT, power brownout | 1) Rate-limit notifications (<=10/sec) 2) Lower LED brightness 3) Check battery voltage under load 4) Reduce MTU assumptions | Stable connection for 2 minutes |
| Button presses don’t register (or double-trigger) | No pull-up, bounce, wrong GPIO | 1) Enable internal pull-up 2) Add 30–60ms debounce 3) Verify GPIO mapping | 20/20 presses reported correctly |
| LEDs flicker / random colors | Power noise, missing ground, too bright, long wires | 1) Lower brightness to <=30% 2) Add 330Ω series resistor on DIN 3) Add 470–1000µF cap across 5V/GND near LEDs 4) Shorten wires | Stable color output |
| Device resets when LEDs animate | Brownout from LED current | 1) Cap brightness 2) Power LEDs from 5V with solid ground 3) Add bulk cap 4) Use a higher C-rate LiPo or USB power for testing | No resets during chase animation |
| Wake word never triggers | Mic not working, wrong PDM pins, threshold too high | 1) Run Arduino mic sanity test 2) Verify GPIO41/42 for PDM 3) Print RMS levels 4) Lower threshold | 18+/20 triggers in quiet room |
| Wake word triggers constantly (false positives) | Threshold too low, noise floor high, mic gain too high | 1) Increase threshold 2) Add cooldown (2s) 3) Reduce mic gain 4) Test in noisy room | <1 false trigger/hour |
| After wake, phone misses first syllable | No pre-roll buffer on iPhone | 1) Add 0.8s ring buffer 2) Start streaming with pre-roll on WAKE_DETECTED | "Hi ESP… status" captures full command |
| Backend transcript is late / laggy | Network, chunk size too large, VAD too conservative | 1) Use 20–40ms audio frames 2) Stop after 1.0s silence 3) Use WebSocket not HTTP chunks | Response within ~1–2s after speaking |
| “Stop” isn’t instant | Stop only handled backend-side | 1) Map gizmo long-press to immediate local stop 2) Phone cancels stream immediately 3) Send backend cancel | Immediate stop (<200ms perceived) |
| Battery % is nonsense | No fuel gauge, divider noise, wrong math | 1) Prefer MAX17048 2) If divider, add 0.1µF cap 3) Use HIGH/MED/LOW buckets | Stable battery indicator |
| MagSafe mount feels weak | Cheap ring, wrong placement, plastic too thick | 1) Use a better ring (Spigen) 2) Ensure ring centered 3) Keep plastic between ring and phone thin 4) Add TPU anti-slip pad | Doesn’t slide during normal use |
| Camera bump prevents flat seating | Pro model camera island interference | 1) Offset ring down a few mm 2) Add a small spacer ridge 3) Adjust enclosure geometry in next STL | Sits flush enough for demo |

## 13) Out of Scope for v1

The following are explicitly **not included** in v1. Do not scope creep.

| Feature | Why Deferred | Target Version |
|---------|--------------|----------------|
| **OTA firmware updates** | Complexity; not needed for demo | v1.5 |
| **Custom "Hey Sociail" wake word** | Requires Espressif customization process | v1.5 |
| **Security hardening** | Open BLE pairing acceptable for demo | v2.0 |
| **Multi-device pairing** | Single phone pairing sufficient | v2.0 |
| **Wi-Fi direct mode** | BLE-only for v1 | v1.5 |
| **Beamforming (2-mic array)** | Single mic sufficient for demo | v2.0 |
| **iOS background wake** | Foreground-only acceptable | v1.5 |
| **Fuel gauge integration** | ADC battery % acceptable | v1.1 |
| **Production enclosure** | 3D print acceptable | v2.0 |

**Tech debt to track:**
1. BLE pairing is open (any phone can connect)
2. No firmware update path (reflash via USB only)
3. No audio encryption over BLE
4. Single-user voice profile only
5. Hardcoded wake word sensitivity (no runtime tuning)
